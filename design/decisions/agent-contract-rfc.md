# RFC: Agent contract adapter hardening

**Status:** accepted (P16, 2026-05)
**Scope:** elevate the three stable adapters (claude-code, codex, generic) from instruction templates to **agent contracts** by adding a `## Agent contract` section to each stable adapter's instruction body. That section names the three axes of the canonical workflow and teaches the v1.6+ surfaces (`--from-file` / `--stdin` / flag-driven `plan brief` & `plan constitution`; configurable protected paths; `write_audit`; `--base-ref`; `--audit-strict`). Adds one diagnostic code, `ADAPTER_CONTRACT_DRIFT`, emitted by `adapter doctor` as a soft signal when a managed file is present but its contract section is missing or out-of-shape.
**Owners:** maintainer
**Related:** [task-readiness-schema](task-readiness-schema-rfc.md) (P10 — declared `writes` + `TASK_WRITES_PROTECTED_PATH`, promoted to the runtime audit this contract teaches) · [governance](governance-rfc.md) (P14 — `LOCK_HELD`, the canonical retry the failure axis names) · [finalization-reconciliation](finalization-reconciliation-rfc.md) (P11 — `task finalize` / `phase reconcile`, the canonical "done" path) · [planning-ux-init-hardening](planning-ux-init-hardening-rfc.md) (P13 — non-interactive `task add`, extended by the non-interactive `plan brief` / `plan constitution` this RFC documents).

## Summary

Through v1.5 the per-agent instruction templates taught only the per-task loop (recommend → context → complete → finalize). v1.6 shipped eleven new surfaces (the `plan brief` / `plan constitution` non-interactive trifecta, `--base-ref`, `--audit-strict`, the `write_audit` envelope, configurable protected paths) that the templates taught **none** of, so a freshly-installed agent reads instruction text that is expired relative to the CLI. This RFC makes the workflow contract **legible and conformance-checkable**: a named `## Agent contract` section, a locked three-axis vocabulary, and a diagnostic for contract-section staleness that is independent of hand-edit drift. code-pact is the control plane other tools' outputs flow into; the contract is what each agent reads on day one.

## Decisions

**1. Add a `## Agent contract` section to each stable adapter's instruction body** covering three axes, each named **verbatim** so a conformance regex can anchor on it (synonyms allowed in body text, the headings are load-bearing):

- **`When to invoke code-pact`** — the canonical command sequence (`recommend` → `task context` → `start` → `block`/`resume` → `complete` → `finalize`), the v1.6 non-interactive bootstrap (`init --non-interactive`; `plan brief` / `plan constitution` with their four modes — TTY wizard, `--from-file`, `--stdin`, flag-driven — and the three-way pairwise mutex), multi-task sequencing (`phase`/`task runbook`, `phase reconcile --write`), and static integrity (`validate`, `plan lint`, `doctor`) run at PR boundaries.
- **`What to verify first`** — `recommend` for tier/budget/planning posture before implementing; read the task's `writes` and `design/rules/protected-paths.md` before declaring writes; run `task finalize --json` (no `--write`) to inspect `data.write_audit` before flipping; in CI pair `--audit-strict` with `--base-ref <default-branch>` so the audit compares against the merge-base (without `--base-ref` it only sees uncommitted changes and `TASK_WRITES_AUDIT_DECLARED_UNUSED` fires); drop `--base-ref` for local pre-commit review.
- **`How to handle failures`** — `LOCK_HELD` → wait/retry (inspect `data.lock_holder`); `VERIFICATION_FAILED` → fix the failing `verification.commands`, re-run `task complete` (idempotent); `TASK_FINALIZE_NOT_ELIGIBLE` → route via `task complete` first; `WRITES_AUDIT_STRICT_FAILED` → fix declared writes (preferred) OR drop `--audit-strict` and document why; `CONFIG_ERROR` → structural argument problem, re-read the surface.

**Rationale:** the contract content is what the agent reads, so it lives in the instruction body by definition; a named anchor lets both a conformance test and a human reader find it. Locking the heading strings gives future doc/template authors a shared lexicon.

**2. Add `ADAPTER_CONTRACT_DRIFT` to `KNOWN_CODES.adapter`** — emitted by `adapter doctor` (single agent) and global `doctor` (every installed agent) when a managed file is present but its `## Agent contract` section is missing or its three sub-headings are incomplete.

**Rationale:** it detects a managed file that exists and has **not** been hand-edited (sha256 matches the manifest) but was generated by a pre-P16 code-pact and lacks the section — a case none of `ADAPTER_FILE_MISSING` / `ADAPTER_FILE_DRIFT` / `ADAPTER_SCHEMA_DRIFT` / `ADAPTER_MANIFEST_*` covers.

**3. Keep it a textual section, not a manifest field or separate file.** **Rationale:** the contract is the body the agent reads; a manifest field would carry only hashes (the conformance test must inspect the body anyway) and a separate file would split the instruction surface in two.

## Diagnostic contract — `ADAPTER_CONTRACT_DRIFT`

- **Severity: warning (advisory).** Surfaces in `data.issues[]` only; does **not** change `adapter doctor`'s ok/exit shape — the existing `ADAPTER_FILE_DRIFT` and friends still own the exit-relevant signal.
- **Detection, per managed file:** locate the literal `## Agent contract` (case-sensitive). Absent → emit with `data.kind = "section_missing"`. Present but fewer than three of the locked sub-headings (`When to invoke code-pact`, `What to verify first`, `How to handle failures`) → emit with `data.kind = "axes_incomplete"` + `data.missing_axes: string[]`.
- **Separate code, not an `ADAPTER_FILE_DRIFT` extension.** `ADAPTER_FILE_DRIFT` means "sha256 differs from the manifest → file was hand-edited" (`--accept-modified` governs it). `ADAPTER_CONTRACT_DRIFT` fires when sha256 **matches** (no hand edit) but the section is absent. The two file-state classifications are independent: conflating them would force a choice between `--accept-modified` (keep edits, ignore contract drift) and overwrite (lose edits to regain the section). Split, `adapter upgrade --write` can reinstate the section while preserving user edits.
- **Recovery:** `adapter doctor --json` → see the code; `adapter upgrade <agent> --write --json` (add `--accept-modified` if the file is hand-edited too). Existing v0.9 upgrade machinery; no new flag.

## Contract surface (additive)

- `KNOWN_CODES.adapter` gains exactly one entry (`ADAPTER_CONTRACT_DRIFT`, warning). `KNOWN_CODES.public` is **unchanged**.
- Every existing command's human-mode and `--json` envelope is unchanged; the new code is an additive `data.issues[]` entry, so `ok: true` consumers keep passing.
- Adapter manifest schema unchanged — no new field, no `schema_version` bump. Instruction file path, role, and selection logic unchanged.
- **Locale:** the three axis HEADINGS are locked in English even in ja-JP variants (load-bearing for the conformance regex); body text under each heading is localised normally.
- **Conformance (stable adapters only):** the existing assertions extend with the literal `## Agent contract` heading, the three verbatim sub-headings, and at least one mention each of `--audit-strict`, `--from-file`, `--stdin`, `write_audit`. `expected-files.txt` fixtures are unchanged (paths unchanged; only body content). cursor / gemini-cli stay excluded — experimental, same posture as the existing suite.

## Alternatives considered

- **Don't ship; let agents read CLI docs directly** — rejected; the instruction file is the single day-one entry point we control, CLI docs are out of scope for "what next?".
- **Embed the entire CLI surface in the instruction file** — rejected as bloat; the section is intentionally short, depth lives behind the doc link.
- **Structured manifest field / separate per-agent contract file** — rejected; see Decision 3 (content lives in the body; splitting the surface forces agents and `adapter doctor` to track two states).
- **Extend `ADAPTER_FILE_DRIFT` instead of a new code** — rejected; it conflates "hand-edited" with "stale template" and breaks the keep-edits-AND-get-the-section path.
- **Wait for cursor / gemini-cli to stabilise before standardising** — rejected; the stable set is closed at three, waiting on experimental upstreams defers a real win.

## Deferred / open questions

- **Auto-injection** of the section on `adapter upgrade --write` over an `--accept-modified` file (surgically add the missing section without overwriting other edits) — v1.7 ships the diagnostic; the surgical mechanism waits for false-positive/negative data.
- **Versioned contract schema** (`agent_contract_v: 1`) — premature with one contract version.
- **cursor / gemini-cli conformance** — deferred to a phase that promotes either adapter to stable.
- No open questions at acceptance: the diagnostic name, axis heading strings, and three-axis taxonomy were settled during drafting. Implementation issues open a follow-up amendment per the P14 lifecycle precedent.
