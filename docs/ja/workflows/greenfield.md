# ゼロから始める（greenfield）ワークフロー

> 🌐 English version: [Greenfield workflow](../../workflows/greenfield.md)

このガイドは、**まだプロダクションコードが存在しない**プロジェクト向けです。空の git リポジトリ（または足場だけのリポジトリ）から始めて、初日から `code-pact` でビルドを構造化したい場合に読んでください。

既存のコードベースに `code-pact` を導入する場合は、代わりに [既存に導入 (brownfield)](brownfield-feature.md) を読んでください。コマンド列そのものは [はじめに](../getting-started.md) にあります。このドキュメントが扱うのは、**プロンプトに何を書くか**、**最初のフェーズをどう形づくるか**、**greenfield プロジェクトに合う導入経路はどれか**です。

## どの導入経路を選ぶか

| こんなとき | 使う経路 |
| --- | --- |
| 使い捨てリポジトリでインストールが動くか確認したいだけ | 経路1 — チュートリアル |
| 最初の1〜2フェーズの明確なイメージが既にある | 経路2 — 手動 |
| brief をもとに AI エージェントにロードマップ全体を起案させたい | 経路3 — AI 支援 |

各経路の詳細は [はじめに](../getting-started.md) を参照してください。スコープが小さくない多くの greenfield プロジェクトでは、**AI 支援が最もレバレッジの高い経路**です。エージェントに brief と constitution を渡せば、ひと通しでフェーズ分解を出力し、`phase import` がその YAML を決定論的に取り込みます。

## brief と constitution に何を書くか

プロジェクト開始時に一度だけ実行する2つの対話ウィザードが、以降のすべてのプロンプトとコンテキストパックが参照する土台を書き出します。ここを**具体的な内容**で埋めることが、AI 支援の経路で「ありきたりな足場」ではなく「役に立つロードマップ」を生む鍵です。

```sh
code-pact plan brief         # → design/brief.md
code-pact plan constitution  # → design/constitution.md
```

`plan brief` は3つの質問をします。**何を作るか**、**誰のためか**、**何が違うのか**。具体的に答えてください。「〜する CLI ツール」は「開発者向けツール」に勝ります。具体的なユーザー像は「開発者」に勝ります。本物の差別化要因は曖昧なものに勝ります。

`plan constitution` は、説明文に加えて**コア原則**をカンマ区切りで集めます。原則は願望ではなく、エージェントが交渉の余地なしとして扱うべき制約です。実際に振る舞いを縛る例:

- *「すべての公開サーフェスは `--json` で JSON のみを stdout に出す」*
- *「この CLI から LLM API を呼ばない」*
- *「`design/` と `.code-pact/state/` 配下のファイルはすべてアトミックに書き込む」*

「きれいなコードを書く」「使いやすくする」のような原則は避けてください。どの判断も変えないからです。

## 最初のフェーズをどう形づくるか

greenfield プロジェクトでは通常、**まず基盤フェーズ**、次に**外部から観測できる能力ごとに1フェーズ**、そして**v1 を切る前に1つの安定化フェーズ**、という形が望ましいです。このパターンは `code-pact` 自身のロードマップ（`P1 基盤` → `P2..P7 段階的な能力` → `P8 安定した制御層`）と一致し、`plan prompt` 経由でプロンプトを与えたとき多くの AI エージェントが生成する形でもあります。

| フェーズの形 | 典型的な中身 |
| --- | --- |
| **基盤 (P1)** | プロジェクトの足場、コアスキーマ、全サブシステムを一度ずつ通す最小のエンドツーエンド骨格。`type: architecture` のタスクが多め。 |
| **能力フェーズ (P2..Pn)** | まとまった能力ごとに1フェーズ。1タスク=1PR のパターンが現実的になるようスコープする。`type: feature` が中心。 |
| **安定化 (Pn+1)** | カバレッジの補完、公開契約の凍結、移行ドキュメント。新機能はなし。`type: test` と `type: docs` が中心。 |

weight は予算ではなく注釈付きの見積りです。1フェーズあたり 5〜30 が典型的な幅です。最初の一巡ではフェーズ間で weight を均等にしても構いません。

## サンプルフェーズを残すべきとき

`code-pact init --sample-phase` は、オプトインの**サンプルフェーズ**（`TUTORIAL`、v1.4+）を書き出します。その id は予約されているので、実際の `P1` と衝突しません。サンプルフェーズが残っていても、生成したロードマップの取り込みは安全です。とはいえ、**`plan prompt` + `phase import` で実際のフェーズができたら削除してください**。これはチュートリアル専用の足場であり、ロードマップの一部ではありません。残す / 改名 / 削除の判断の全体は [concepts/sample-phase.md](../../concepts/sample-phase.md)（英語）を参照してください。ループの動きを見たいだけなら、`code-pact tutorial` の方が適切です（あとに何も残しません）。

## 推奨される初回 PR

greenfield プロジェクトで最小かつ有用な初回 PR は、**verify コマンドをエンドツーエンドで通す P1 の1タスク**です。これにより次が確認できます:

- `code-pact init` が妥当な `design/` ツリーと `.code-pact/project.yaml` を生成したこと。
- `adapter install` が、エージェントが実際に読める `CLAUDE.md` / `AGENTS.md` / agent-instructions.md を生成したこと。
- `task complete` が `pnpm test`（または選んだ verify コマンド）を実行し、`done` イベントを書き込むこと。

この1往復が済めば、ロードマップの残りは機械的に進みます。
